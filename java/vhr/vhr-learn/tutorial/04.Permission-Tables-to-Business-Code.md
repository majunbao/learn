# 04. Permission Tables to Business Code

## In One Sentence

Wire the 5 core permission tables in the vhr database to Java code, write a simple query endpoint, and see how data flows through the layers.

## Core Permission Tables

vhr's permission system consists of 5 tables:

| Table       | Description              |
| ----------- | ------------------------ |
| `hr`        | Operator table (users)   |
| `role`      | Role table               |
| `menu`      | Menu / permission table  |
| `hr_role`   | Operator-role join table |
| `menu_role` | Role-menu join table     |

Relationship diagram:

```
hr ←→ hr_role ←→ role ←→ menu_role ←→ menu
```

## Step 1: Create the Directory Structure

Create the following directories in your project (in this order):

1. Under `com.vhr`, create four sibling directories: `bean`, `mapper`, `service`, `controller`
2. Inside `service`, create the `impl` subdirectory

Final structure:

```
src/main/java/com/vhr
├── bean          # Entity classes
├── mapper        # Mapper interfaces
├── service       # Service layer interfaces
│   └── impl      # Service layer implementations
└── controller    # Controller layer
```

### Layer Comparison

| Java Spring Boot  | Node.js (Express/Koa)           | Vue                    | Purpose                                  |
| ----------------- | ------------------------------- | ---------------------- | ---------------------------------------- |
| `bean` / `entity` | `model` / `entity`              | `data` / component `props` | Define data structures, mirror database tables |
| `mapper`          | `dao` / `repository`            | -                      | Write SQL, interact with the database    |
| `service`         | `service`                       | `store` / composables  | Write business logic, process data       |
| `controller`      | `controller` / `router handler` | -                      | Receive requests, call service, return response |

## Step 2: Write Entity Classes (Bean)

> Note: "bean" here is a directory name meaning "entity class". "Spring Bean" (capital B) is a Spring-managed object — related but different concepts! The entity classes we're writing now aren't yet managed by Spring, so they don't need `@Component`.

## Step 3: Write Mapper Interface and XML

### 1. MenuMapper Interface

Create `MenuMapper.java` under `com.vhr.mapper`:

```java
package com.vhr.mapper;

import com.vhr.bean.Menu;
import org.apache.ibatis.annotations.Mapper;

import java.util.List;

@Mapper
public interface MenuMapper {
    List<Menu> getAllMenus();
}
```

### 2. MenuMapper.xml

First, create a `mapper` directory under `src/main/resources`, then create `MenuMapper.xml` inside it:

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper
        PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.vhr.mapper.MenuMapper">

    <resultMap id="BaseResultMap" type="com.vhr.bean.Menu">
        <id property="id" column="id"/>
        <result property="url" column="url"/>
        <result property="path" column="path"/>
        <result property="component" column="component"/>
        <result property="name" column="name"/>
        <result property="iconCls" column="iconCls"/>
        <result property="keepAlive" column="keepAlive"/>
        <result property="requireAuth" column="requireAuth"/>
        <result property="parentId" column="parentId"/>
        <result property="enabled" column="enabled"/>
    </resultMap>

    <select id="getAllMenus" resultMap="BaseResultMap">
        SELECT * FROM menu WHERE enabled = 1
    </select>

</mapper>
```

#### Mapper XML Explained

| Part                                    | Description                                                                 | Node.js Equivalent                                 |
| --------------------------------------- | --------------------------------------------------------------------------- | -------------------------------------------------- |
| `namespace="com.vhr.mapper.MenuMapper"` | Links to the Java interface `MenuMapper`; MyBatis uses this to find which interface to implement | -                                                  |
| `<resultMap>`                           | Defines the mapping between database column names and Java entity property names | Similar to an ORM model definition (Sequelize / TypeORM) |
| `<select id="getAllMenus">`             | The `id` must match the method name in the Java interface; MyBatis automatically converts the SQL result into Java objects | Equivalent to `dao.getAllMenus()` in Node.js       |

#### Quick Comparison with Node.js

**Node.js approach (e.g. using mysql2):**

```javascript
// dao/menuDao.js
async function getAllMenus() {
  const [rows] = await connection.execute('SELECT * FROM menu WHERE enabled = 1');
  // Manually map database results to objects
  return rows.map(row => ({
    id: row.id,
    url: row.url,
    // ... manually map every field
  }));
}
```

**MyBatis approach (the XML above):**

```xml
<select id="getAllMenus" resultMap="BaseResultMap">
    SELECT * FROM menu WHERE enabled = 1
</select>
```

The difference: MyBatis uses `<resultMap>` to automatically map database columns to Java objects — no manual row-by-row mapping needed.

### 3. Update application.properties with MyBatis-Plus Configuration

Open `src/main/resources/application.properties` and append:

```properties
# ================================
# MyBatis-Plus
# ================================
# Location of MyBatis Mapper XML files
mybatis-plus.mapper-locations=classpath:mapper/*.xml
# Print executed SQL in console for debugging
mybatis-plus.configuration.log-impl=org.apache.ibatis.logging.stdout.StdOutImpl
```

> **What do these do?**
> 1. `mybatis-plus.mapper-locations`: Tells MyBatis where to find our `MenuMapper.xml` file
> 2. `mybatis-plus.configuration.log-impl`: Prints the executed SQL statements in the IDEA console, which makes debugging easier

## Step 4: Write the Service Layer

### 1. MenuService Interface

Create `MenuService.java` under `com.vhr.service`:

```java
package com.vhr.service;

import com.vhr.bean.Menu;

import java.util.List;

public interface MenuService {
    List<Menu> getAllMenus();
}
```

### 2. MenuServiceImpl Implementation

Create `MenuServiceImpl.java` under `com.vhr.service.impl`:

```java
package com.vhr.service.impl;

import com.vhr.bean.Menu;
import com.vhr.mapper.MenuMapper;
import com.vhr.service.MenuService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

import java.util.List;

@Service
public class MenuServiceImpl implements MenuService {

    @Autowired
    private MenuMapper menuMapper;

    @Override
    public List<Menu> getAllMenus() {
        return menuMapper.getAllMenus();
    }
}
```

#### Service Layer Explained

##### What is a Bean?

A Bean is simply **an object managed by Spring**.

**Compared to Node.js:**

```javascript
// In Node.js we use objects like this
const menuService = require('./menuService'); // must manually import every time
menuService.getAllMenus();
```

**In Spring:**

```java
@Autowired
private MenuService menuService; // no need to new — Spring creates it for us
menuService.getAllMenus();
```

**Compared to Vue:**
A Bean is somewhat like a globally registered Vue component or a Pinia store — you register it once and can use it anywhere without recreating it.

| Java Spring Bean                          | Node.js Equivalent               | Vue Equivalent                         |
| ----------------------------------------- | -------------------------------- | -------------------------------------- |
| `@Service` / `@Component` / `@Controller` | `module.exports`                 | Vue plugin global registration / Pinia store |
| `@Autowired` injection                    | `require()` / `import`           | `useStore()` or `inject`               |

| Part         | Description                                                                                             | Node.js Equivalent                                |
| ------------ | ------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| `@Service`   | Tells Spring "manage this class, register it as a Bean"; we can then get it anywhere via `@Autowired`   | -                                                 |
| `@Autowired` | Auto-injects the dependency; Spring creates a `MenuMapper` instance and assigns it to `menuMapper`      | Equivalent to `require('./menuDao')` in Node.js   |
| `@Override`  | Indicates this method implements the method declared in the `MenuService` interface                     | -                                                 |

#### Service Layer vs. Node.js

**Node.js approach:**

```javascript
// service/menuService.js
const menuDao = require('../dao/menuDao'); // equivalent to @Autowired

async function getAllMenus() {
  // Business logic goes here: data filtering, transformation, etc.
  const menus = await menuDao.getAllMenus();
  return menus;
}

module.exports = { getAllMenus };
```

**Java approach (the code above):**

```java
@Service
public class MenuServiceImpl implements MenuService {
    @Autowired
    private MenuMapper menuMapper; // equivalent to Node.js require import

    @Override
    public List<Menu> getAllMenus() {
        return menuMapper.getAllMenus();
    }
}
```

## Step 5: Write the Controller Layer

Create `MenuController.java` under `com.vhr.controller`:

```java
package com.vhr.controller;

import com.vhr.bean.Menu;
import com.vhr.service.MenuService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import java.util.List;

@RestController
public class MenuController {

    @Autowired
    private MenuService menuService;

    @GetMapping("/menu/all")
    public List<Menu> getAllMenus() {
        return menuService.getAllMenus();
    }
}
```

#### Controller Layer Explained

| Part                         | Description                                                                                                                          | Node.js / Vue Equivalent                              |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| `@RestController`            | Tells Spring this is a Controller and all return values are automatically converted to JSON (equivalent to `@Controller + @ResponseBody`) | Equivalent to a route handler in Express / Koa        |
| `@GetMapping("/menu/all")`   | Defines the GET request path; visiting `/menu/all` in the browser calls this method                                                  | Equivalent to `app.get('/menu/all', handler)` in Express |
| `@Autowired`                 | Auto-injects `MenuService`                                                                                                           | Equivalent to `require('./menuService')` in Node.js   |

#### Controller Layer vs. Node.js

**Node.js Express approach:**

```javascript
// controller/menuController.js
const menuService = require('../service/menuService');
const express = require('express');
const router = express.Router();

router.get('/menu/all', async (req, res) => {
    const menus = await menuService.getAllMenus();
    res.json(menus); // return JSON data
});

module.exports = router;
```

**Java Spring Boot approach (the code above):**

```java
@RestController
public class MenuController {
    @Autowired
    private MenuService menuService; // equivalent to Node.js require import

    @GetMapping("/menu/all") // equivalent to Express router.get
    public List<Menu> getAllMenus() {
        return menuService.getAllMenus();
        // Spring automatically converts List<Menu> to JSON — no manual res.json needed
    }
}
```
